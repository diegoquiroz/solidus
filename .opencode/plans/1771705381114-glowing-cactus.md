## What “parity” means for this wave

1. Customer identity semantics match Pay:
   - owner identity in `ownerType` / `ownerId`
   - processor identity in `processor` / `processorId`
   - row `id` is opaque/surrogate (not semantic)
2. Checkout owner linking matches Pay behavior:
   - `checkout.session.completed` and `checkout.session.async_payment_succeeded` follow identical linking/sync behavior
   - malformed/unknown `client_reference_id` is best-effort no-op (do not fail webhook pipeline)
3. Customer lifecycle parity:
   - `customer.updated` also reconciles default payment method state
   - `customer.deleted` performs strong cleanup behavior (not just shallow customer upsert)
4. Runtime and persistence contracts are aligned for Stripe paths (core contracts, adapters, schema artifacts, tests, docs).

## Phase 7 (Optional) - Pay-style model ergonomics

**Goal**: Achieve Pay gem ergonomics where billing methods are available directly on owner models (e.g., `user.payCustomer().charge()`) instead of requiring `facade.convenience` calls.

**Pay pattern reference**:
- Pay uses `pay_customer` class macro that mixes in `CustomerExtension`
- Owner model gets `payment_processor` accessor that returns processor-specific customer instance
- Methods like `charge()`, `subscribe()`, `add_payment_method()` available directly on the customer instance
- File: `/Users/diego/Developer/clones/pay/lib/pay/attributes.rb` (lines 106-119)

**Feasibility in Solidus**:

YES - Can be achieved with minimal changes:

1. **Create global facade singleton** (`src/packages/core/global-facade.ts`):
   - Allow one-time configuration: `Solidus.configure({ stripe, repositories, ... })`
   - Expose global accessor: `Solidus.facade` or `getGlobalFacade()`
   - Maintains DI flexibility while enabling ergonomic access

2. **Create model mixin helper** (`src/packages/core/model-extension.ts`):
   ```ts
   // Usage in user's model class
   class Workspace extends Model {
     static billing = solidusBillingMixin({
       ownerType: "Workspace",
       getOwnerId: (instance) => instance.id,
     });
   }
   
   // Then use ergonomically
   const workspace = await Workspace.findByPk(id);
   await workspace.billing.setProcessor();  // calls facade.convenience.setOwnerStripeProcessor
   await workspace.billing.charge({ amount: 1000, currency: "usd" });  // facade.api.charges.charge
   await workspace.billing.subscribe({ priceId: "price_monthly" });  // facade.api.subscriptions.create
   ```

3. **Implementation approach**:
   - Add `solidusBillingMixin()` factory that returns a getter function
   - Getter returns proxy object with methods bound to owner context
   - Methods delegate to global facade with ownerType/ownerId pre-filled
   - Lazy-loads processor customer ID from ownerCustomers repository
   - Minimal runtime overhead, type-safe with generics

**Trade-offs**:
- Requires global configuration (loses per-instance DI flexibility)
- Must be opt-in (don't force singleton pattern on all users)
- Works best with ORM models (Sequelize/Mongoose), harder with plain objects
- Adds small API surface (mixin factory + global config)

**Files to create/modify**:
- `src/packages/core/global-facade.ts` - global configuration and accessor
- `src/packages/core/model-extension.ts` - mixin factory
- `src/packages/core/index.ts` - re-exports
- Add to docs: usage examples for Sequelize models

**Alternative approach** (if global singleton is unacceptable):
- Pass facade reference to model instances via context/dependency injection
- Mixin accepts facade factory: `solidusBillingMixin({ facade: () => app.facade })`
- More flexible but requires manual wiring in each model

### Key Architectural Decisions

1. **No global singleton (yet)**: The facade requires explicit DI (stripe client, repositories). Phase 7 proposes adding optional global singleton for ergonomics.
2. **Queue adapter pattern**: Background jobs use `QueueAdapter` interface, not ActiveJob. Options: DB outbox (prod), inline (dev), or bring-your-own (BullMQ/SQS).
3. **Identity semantics**: 
   - `ownerType`/`ownerId` = your app's entity (Workspace, User, etc.)
   - `processor`/`processorId` = Stripe's identifiers
   - Record `id` = opaque UUID (not derived from other fields)
4. **Checkout linking**: Only works with format `<ModelName>_<OwnerId>` (e.g., `Workspace_42`). Malformed/unknown references are silently ignored (Pay behavior).
